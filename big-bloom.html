Big Bloom Financial Software

Introduction

I was contacted by members of the Market Data Company team to help design the beta release of a desktop application currently in development.

This application seeks to address the problem of cost control for a widely used financial data software platform.

The Cost Control Problem

Companies can find it difficult to keep costs under control when using financial data software¹.

Causes of this problem can include:

  • Complex pricing schedules
  • Cross departmental usage
  • Confusing Monthly cost/usage reports
  • Periodic requests do not automatically turn off
Prototype Usability Issues

The Market Data Team had attempted to tackle the problem prior to me joining with a prototype of their intended product.

Built using Visual Studio 2010 (Windows Forms), the prototype (shown above) had usability issues that they were hoping to address at the Beta stage.

Understanding Different User Needs

It became clear that there were two primary user groups: analysts and managers.

The beta release would address the needs of both analyst and manager users. Key takeaways from the user research stage:

  • Include a dashboard with summary visualizations
  • Allow user to assign costs to different departments
  • Minimize steps needed to accomplish goals in application
Constraints

The development environment was chosen to be Visual Studio 10 (.NET 4), therefore the interface would have to be designed using the same Windows Form controls that were utilized for the prototype².

Limited font selection, static-size windows, default buttons and controls, as well as a lack of alignment tools added significant obstacles to the design process.

Defining Clear Principles

Taking into account both the user analysis insights and the constraints of the design environment, a set of principles were recorded:

  1. Feature Exposure
  2. Universal Language
  3. Optimize
  4. Visual Organization
User Workflow

An analyst must collect a monthly usage report from the financial tool, and load that report into Big Bloom. This updates the internal usage data.

The analyst can then export a spreadsheet from the Big Bloom tool. This sheet will contain analysis that points to potential cost savings opportunities.

The core of the Big Bloom application is focused around this load/export function.

Wireframing

Multiple iterations of the app were designed for review and feedback.

The goal here was to design a workflow that upheld the design principles, and that the partners (who acted as proxy usability testers) found intuitive.

Beta Version

The beta version was meant to be demonstrated to potential clients, and to gauge the value add of the tool’s functionality. On-site feedback pointed to a strong interest in the dashboard, as well as the automated analysis captured in the report output.

The next steps will be to migrate the platform to a web-based interface. In doing so, more freedom will be granted to design decisions involving visuals, layout, and interactions.
